# Kiro Memory System

## Overview

Kiro is an AI-native IDE from Amazon (built on a Code OSS fork) that emphasizes spec-driven development. It evolved from the Amazon Q Developer ecosystem but is a separate product. Kiro's "memory" is implemented through Steering files (persistent markdown instructions) and Specifications (structured development workflows). It also maintains a project context memory that carries across sessions.

## Storage Locations

| Component | Path | Purpose |
|-----------|------|---------|
| Steering files | `.kiro/steering/` or `.amazonq/rules/` | Persistent instructions |
| Specifications | `.kiro/specs/<feature>/` | Structured dev workflow artifacts |
| Product overview | `.kiro/steering/product.md` | Product purpose and goals |
| Tech stack | `.kiro/steering/tech.md` | Frameworks, libraries, constraints |
| Project structure | `.kiro/steering/structure.md` | File organization, naming, architecture |
| Spec requirements | `.kiro/specs/<feature>/requirements.md` | User stories, acceptance criteria |
| Spec design | `.kiro/specs/<feature>/design.md` | Technical architecture |
| Spec tasks | `.kiro/specs/<feature>/tasks.md` | Implementation plan with checkboxes |
| Codebase index | Local (in-memory/disk) | Indexed codebase for retrieval |

## Memory Types

### Steering Files (Persistent Instructions)

Markdown files that give Kiro persistent knowledge about your workspace. Three foundation files are recommended:

1. **product.md**: Product purpose, target users, key features, business objectives. Helps Kiro understand the "why" behind technical decisions.
2. **tech.md**: Frameworks, libraries, development tools, technical constraints. Kiro prefers your established stack over alternatives.
3. **structure.md**: File organization, naming conventions, import patterns, architectural decisions. Ensures generated code fits the existing codebase.

Additional steering files can be created for specific concerns (testing, security, API conventions, etc.).

**Legacy path**: `.amazonq/rules/` is still supported from the Amazon Q Developer era. Kiro recognizes both paths.

### Specifications (Spec-Driven Development)

Kiro's most distinctive feature. Specs transform feature ideas into structured implementation plans through three phases:

1. **Requirements** (`requirements.md`): User stories with acceptance criteria in EARS notation (Event-Action-Response-State). Defines *what* needs to be built.
2. **Design** (`design.md`): Technical architecture and implementation approach. Defines *how* it will be built.
3. **Tasks** (`tasks.md`): Discrete, trackable implementation steps as a checklist. The agent works through tasks sequentially.

Specs persist in the repo and serve as both documentation and agent memory for ongoing feature work.

### Project Context Memory

Kiro maintains cross-session project context, carrying memory of previous interactions. This enhances continuity when returning to a project.

### Codebase Index

Default index size is 250 MB. The index supports retrieval-augmented context for code generation. Users can manually adjust the size limit.

## Persistence Scope

- **Steering files**: Persist in the repo. Version-controllable. Shared via git.
- **Specifications**: Persist in the repo. Version-controllable. Serve as living documentation.
- **Project context**: Persists across sessions (mechanism not fully documented).
- **Codebase index**: Rebuilt as needed. Not persisted across reinstalls.

## User Control

- Steering files are directly editable markdown.
- Specs can be generated by Kiro or written manually.
- Users can modify any phase of a spec independently.
- Task checkboxes track progress; the agent respects completed/incomplete state.
- No auto-memory approval UI -- steering files and specs are the primary persistence mechanism.

## Implementation Details

- Built on Code OSS (VS Code fork), so the extension ecosystem is partially compatible.
- Kiro and Amazon Q Developer are separate products. Q Developer is an IDE plugin; Kiro is a standalone IDE.
- The `.amazonq/rules/` path provides migration continuity from Q Developer.
- Spec-driven development is the core differentiator: rather than chat-based ad-hoc coding, Kiro structures work into requirements -> design -> tasks.
- EARS notation for acceptance criteria provides machine-parseable requirement structure.
- Kiro CLI also supports steering via markdown files.

## Strengths

- **Spec-driven development is unique**: The three-phase spec workflow (requirements -> design -> tasks) provides structured memory that doubles as documentation.
- **Specs are version-controllable**: Living documents that evolve with the feature and persist in the repo.
- **Foundation steering files**: product.md / tech.md / structure.md is a clean, well-defined instruction taxonomy.
- **Cross-session context**: Project context memory carries forward without manual setup.
- **Migration path from Q Developer**: `.amazonq/rules/` compatibility eases transition.

## Weaknesses

- **No auto-memory from sessions**: Kiro doesn't automatically persist learnings from coding sessions. All persistent knowledge requires steering files or specs.
- **Spec overhead**: For small changes, the three-phase spec workflow adds ceremony. Not everything needs requirements -> design -> tasks.
- **Young ecosystem**: As a newer product (post-Q Developer), community resources and third-party integrations are still developing.
- **Separate from Q Developer**: Users of Q Developer in existing IDEs don't get Kiro's spec features without switching editors.
- **Limited memory documentation**: Cross-session project context is mentioned but not well-documented technically.
- **250 MB index limit**: Large monorepos may exceed the default codebase index size.
